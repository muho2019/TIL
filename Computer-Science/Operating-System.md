# 운영체제

운영체제도 하나의 프로그램입니다. 따라서 운영체제 또한 여느 프로그램과 마찬가지로 메모리에 적재되어야 합니다.

하지만 운영체제는 특별한 프로그램이기 때문에 **커널 영역**(kernel space)이라는 공간에 적재되어 실행됩니다. 이 커널 영역을 제외한 나머지 영역을 **사용자 영역**(user space)이라고 합니다. 바로 이 영역에 우리가 사용하는 각종 응용 프로그램이 적재됩니다.

운영체제는 프로그램이 실행되어 메모리에 적재될 때 어떤 프로그램을 어느 메모리에 적재할 지 정하고 종료된 프로그램을 메모리에서 제거하는 역할을 합니다. 프로그램이 실행되는 데 필수 요소인 CPU에 프로그램을 할당하는 것 역시 운영체제가 담당합니다.

## 커널 (Kernel)

커널은 자원에 접근하고 조작하는 기능, 프로그램이 올바르고 안전하게 실행되게 하는 기능 등의 운영체제 핵심 서비스를 담당합니다.

## 이중모드 (Dual Mode)

운영체제는 사용자가 실행하는 응용 프로그램이 자원에 직접 접근하는 것을 방지하여 자원을 보호합니다. 만약 응용 프로그램이 CPU, 메모리, 하드 디스크 등에 마음대로 접근하고 조작할 수 있다면 자원이 무질서하게 관리될 것이고, 응용 프로그램에서 발생한 아주 작은 실수로 인해 컴퓨터 전체에 악영향을 끼칠 수 있습니다.

따라서 운영체제는 응용 프로그램들이 자원에 접근하려고 하면 오직 자신을 통해서만 접근하도록하여 자원을 보호합니다. 응용 프로그램의 요청을 받은 운영체제는 응용 프로그램 대신 자원에 접근하여 요청한 작업을 수행합니다.

이러한 운영체제의 문지기 역할은 이중 모드로써 구현됩니다. **이중 모드**란 CPU가 명령어를 실행하는 모드를 크게 **사용자 모드**(user mode)와 **커널 모드**(kernel mode)로 구분하는 방식입니다. CPU는 명령어를 사용자 모드로써 실행할 수 있고, 커널 모드로써 실행할 수 있습니다.

사용자 모드는 운영체제 서비스를 제공받을 수 없는 실행 모드입니다. 즉, 커널 영역의 코드를 실행할 수 없는 모드입니다. 일반적인 응용 프로그램은 기본적으로 사용자 모드로 실행됩니다. 사용자 모드로 실행 중인 CPU는 입출력 명령어와 같이 하드웨어 자원에 접근하는 명령어를 실행할 수 없습니다.

커널 모드는 운영체제 서비스를 제공받을 수 있는 실행 모드입니다. 즉, 커널 영역의 코드를 실행할 수 있는 모드입니다. CPU가 커널 모드로 명령어를 실행하면 자원에 접근하는 명령어를 비롯한 모든 명령어를 실행할 수 있습니다. 운영체제는 커널 모드로 실행되기 때문에 자원에 접근할 수 있습니다.

## 시스템 호출 (System Call)

사용자 모드로 실행되는 프로그램이 자원에 접근하는 운영체제 서비스를 제공받으려면 운영체제에 요청을 보내 커널 모드로 전환되어야 합니다. 이때 운영체제 서비스를 제공받기 위한 요청을 **시스템 호출**이라고 합니다. 사용자 모드로 실행되는 프로그램은 시스템 호출을 통해 커널 모드로 전환하여 운영체제 서비스를 제공받을 수 있습니다.

시스템 호출은 일종의 인터럽트입니다. 정확히는 소프트웨어적인 인터럽트입니다. 인터럽트는 입출력장치에 의해 발생하기도 하지만 인터럽트를 발생시키는 특정 명령어에 의해 발생하기도 하는데, 이를 소프트웨어 인터럽트라고 합니다.

시스템 호출을 발생시키는 명령어가 실행되면 CPU는 지금까지의 작업을 백업하고 커널 영역 내에 시스템 호출을 수행하는 코드를 실행한 뒤 다시 기존에 실행하던 응용 프로그램으로 복귀하여 실행을 계속해 나갑니다.

## 프로세스

프로세스는 사용자가 볼 수 있는 공간에서 실행되는 프로세스인 **포그라운드 프로세스**(foreground process)와 사용자가 보지 못하는 뒤편에서 실행되는 프로세스인 **백그라운드 프로세스**(background process)가 있습니다.

백그라운드 프로세스 중에는 사용자와 상호작용하는 프로세스가 있고, 상호작용하지 않는 프로세스가 있습니다. 사용자와 상호작용하지 않는 백그라운드 프로세스를 유닉스 체계의 운영체제에서는 **데몬**(daemon)이라고 부르고, 윈도우 운영체제에서는 **서비스**(service)라고 합니다.

## 프로세스 제어 블록 (PCB, Process Control Block)

프로세스는 실행을 위해 CPU를 필요로 하지만 CPU 자원은 한정적입니다. 그렇기에 프로세스는 차례대로 돌아가며 한정된 시간만큼만 CPU를 이용합니다. 정해진 시간만큼 CPU를 이용하고 시간이 끝나면 타이머 인터럽트가 발생하며 자신의 차례를 양보하고 다음 차례가 올 때까지 기다립니다.

운영체제는 이러한 프로세스의 실행 순서를 관리하고 CPU 자원을 배분합니다. 이를 위해 운영체제는 **프로세스 제어 블록**을 이용합니다.

프로세스 제어 블록은 프로세스와 관련된 정보를 저장하는 자료 구조입니다. **PCB는 커널 영역에 생성됩니다.** 운영체제가 PCB를 통해 프로세스를 식별하고 처리하는 데 필요한 정보를 판단합니다. PCB는 프로세스 생성 시에 만들어지고 실행이 끝나면 폐기됩니다. 즉 '새로운 프로세스가 생성되었다'는 말은 '운영체제가 PCB를 생성했다'라는 말과 같습니다.

PCB에 담기는 정보는 **프로세스 ID**(PID, process ID), **레지스터 값**, **프로세스 상태**, **CPU 스케줄링 정보**, **메모리 관리 정보**, **사용한 파일과 입출력장치 목록**이 있습니다.

- **프로세스 ID**: 프로세스를 식별하기 위해 부여되는 고유번호입니다. 동일한 프로그램을 두 번 실행하면 서로 다른 PID가 부여됩니다.
- **레지스터 값**: 프로세스는 자신이 진행했던 작업들을 저장하고 복원합니다. 그래서 프로세스가 실행하며 사용했던 프로그램 카운터를 비롯한 레지스터 값들이 담깁니다.
- **프로세스 상태**: 현재 프로세스가 입출력장치를 기다리고 있는 상태인지, CPU를 기다리고 있는 상태인지, 아니면 CPU를 이용하고 있는 상태인지 등의 상태 정보가 저장됩니다.
- **CPU 스케줄링 정보**: 프로세스가 언제, 어떤 순서로 CPU를 할당받을지에 대한 정보도 저장됩니다.
- **메모리 관리 정보**: 프로세스가 메모리의 어디에 적재되어 있는지에 대한 정보가 있어야 합니다. 프로세스의 주소를 알기 위하여 베이스 레지스터, 한계 레지스터, 페이지 테이블 정보가 저장됩니다.
- **사용한 파일과 입출력장치 목록**: 프로세스가 실행 중 특정 입출력장치나 파일을 사용하면 PCB에 해당 내용이 명시됩니다.

## 컨텍스트 스위칭 (Context Switching)

CPU를 할당받아 실행되고 있는 프로세스가 시간이 다 되어 다음 프로세스에게 CPU 사용을 양보하면 어떻게 될까요?

기존에 실행되고 있던 프로세스는 프로그램 카운터를 비롯한 각종 레지스터 값, 메모리 정보, 실행을 위해 열었던 파일이나 사용한 입출력장치 등의 중간 정보를 백업합니다. 그래야 다음에 CPU 사용 차례가 왔을 때 이전에 실행하던 내용을 이어서 실행할 수 있습니다. 이러한 중간 정보, 즉 하나의 프로세스 수행을 재개하기 위해 기억해야 할 정보를 **컨텍스트**(문맥, context)라고 합니다. 이 컨텍스트는 해당 프로세스의 PCB에 표현되어 있습니다. PCB에 기록되는 정보들을 컨텍스트라고 봐도 무방합니다.

프로세스의 CPU 사용 시간이 다 되거나 예기치 못한 상황이 발생하여 인터럽트가 발생하면 운영체제는 해당 프로세스의 PCB에 문맥을 백업합니다. 그리고 그 다음 차례의 프로세스의 컨텍스트를 복구합니다. 이렇게 기존 프로세스의 컨텍스트를 PCB에 백업하고, 새로운 프로세스를 실행하기 위해 컨텍스트를 PCB로부터 복구하여 새로운 프로세스를 실행하는 것을 **컨텍스트 스위칭**(context switching)이라고 합니다. 컨텍스트 스위칭은 충분히 빠르기 때문에 우리가 볼 때 여러 프로세스들이 동시에 실행되는 것처럼 보입니다.

## 프로세스 메모리 영역

PCB는 커널 영역에 생성된다고 했습니다. 그리고 프로세스는 사용자 영역에 로드됩니다.

하나의 프로세스는 사용자 영역에 저장될 때 **코드 영역**, **데이터 영역**, **힙 영역**, **스택 영역**으로 나뉩니다.

- **코드 영역(code segment)**: **텍스트 영역**(text segment)이라고도 부릅니다. 여기에는 실행할 수 있는 코드, 즉 기계어로 된 명령어가 저장됩니다. CPU가 실행할 명령어가 담겨 있기 때문에 쓰기가 금지되어 있습니다. 즉, 코드 영역은 읽기 전용 공간입니다. 동일한 명령에 대해서 처음 한 번만 로드되고 모든 프로세스에서 공유됩니다. 코드 영역은 정적 할당 영역입니다.
- **데이터 영역(data segment)**: 전역 변수(global variables)가 저장됩니다. 데이터 영역은 정적 할당 영역입니다.
- **힙 영역(heap segment)**: 프로그램 실행 중에 동적으로 할당되는 영역입니다. 참조형 객체가 저장됩니다. 메모리 주소 공간에서 위쪽으로 할당됩니다.
- **스택 영역(stack segment)**: 함수 호출, 지역 변수(local variables), 함수 호출 시 반환 주소가 저장됩니다. 힙 영역과 마찬가지로 동적 할당 영역입니다. 메모리 주소 공간에서 아래쪽으로 할당됩니다.

## 프로세스 상태

프로세스가 가질 수 있는 상태는 아래와 같습니다.

- **생성 상태(new)**: 프로세스가 생성되는 중인 상태를 나타냅니다. 메모리에 적재되어 PCB를 할당받은 상태를 말합니다.
- **준비 상태(ready)**: CPU에 할당되기를 기다리는 상태입니다.
- **실행 상태(running)**: CPU를 할당받아 실행 중인 상태입니다. 실행 상태인 프로세스는 할당된 일정 시간 동안만 CPU를 사용할 수 있고, 할당된 시간이 다 되면 타이머 인터럽트가 발생하며 준비 상태가 됩니다. 또한 실행 도중 입출력장치를 사용하여 입출력장치의 작업이 끝날 때까지 기다려야 한다면 대기 상태가 됩니다.
- **대기 상태(blocked)**: 입출력장치를 이용한 작업은 CPU 처리 속도에 비해 느리기 때문에 입출력 작업을 요청한 프로세스는 입출력장치가 작업을 끝낼 때까지, 즉 입출력 완료 인터럽트를 받을 때까지 기다려야 합니다. 이 상태를 대기 상태라고 하며, 입출력 작업이 완료되면 준비 상태가 됩니다.
- **종료 상태(terminated)**: 프로세스가 실행을 마친 상태입니다. 프로세스가 종료되면 운영체제는 PCB와 프로스세가 사용한 메모리를 정리합니다. 다만, 부모 프로세스가 `wait()` 시스템 호출을 하기 전까지는 프로세스 테이블에 해당 프로세스의 항목이 남아 있는데, 이 상태의 프로세스를 좀비(zombie) 프로세스라고 부릅니다.

## 프로세스 계층 구조

프로세스는 실행 도중 시스템 호출을 통해 다른 프로세스를 생성할 수 있습니다. 새 프로세스를 생성한 프로세스를 **부모 프로세스**(parent process), 부모 프로세스에 의해 생성된 프로세스를 **자식 프로세스**(child process)라고 합니다. 이러한 구조를 트리 구조로 표현할 수 있는데, 이를 프로세스 계층 구조라고 합니다.

모든 프로세스의 가장 위에 있는 최초의 프로세스는 무엇일까요? 유닉스 운영체제에서는 `init`, 리눅스 운영체제에서는 `systemd`, macOS에서는 `launchd`라고 합니다.

## 프로세스 생성 기법

프로세스 생성은 운영체제가 새로운 실행 흐름을 시작하는 핵심적인 작업입니다.

### 유닉스

부모 프로세스는 `fork()`를 호출하여 자신의 복사본을 만들어 자식 프로세스를 생성합니다. 이때 부모 프로세스의 모든 메모리 영역과 열린 파일 목록, 레지스터 상태 등이 복사됩니다. 그리고 자식의 프로세스 ID를 반환받습니다. 자식 프로세스는 `exec()`를 통해 자신의 메모리 공간을 다른 프로그램으로 교체하여 완전히 새로운 프로세스로 변환합니다. 자식 프로세스가 `exec()`를 호출하지 않는 경우도 있습니다. 이 경우 부모 프로세스와 자식 프로세스는 같은 코드를 병행하여 실행하는 프로세스가 됩니다.

전통적으로 `fork()`는 부모의 메모리 공간을 자식에게 복사하는 방식으로 작동했지만 자식 프로세스 생성 직후 `exec()`를 호출한다는 점을 고려할 때 이러한 복사는 불필요할 수 있습니다. **Copy-on-Write**(COW)는 이러한 비효율성을 개선합니다. 이 기술은 부모와 자식 프로세스가 초기에 동일한 페이지를 공유하도록 허용합니다. 만약 부모나 자식 프로세스 중 하나가 공유된 페이지에 쓰기 작업을 시도하면 그때서야 해당 페이지의 복사본이 생성되어 각 프로세스가 자신만의 독립적인 페이지를 가질 수 있게 됩니다. 이를 통해 프로세스 생성 속도가 빨라지고 필요한 경우에만 메모리 복사 오버헤드가 발생하여 효율적입니다.

### 리눅스

리눅스에서는 `clone()`을 이용하여 프로세스 또는 스레드와 유사한 태스크를 생성합니다. `clone()`에 전달되는 플래그 세트에 따라 생성되는 태스크가 프로세스처럼 동작할지(자원 공유없이) 스레드처럼 동작할지(메모리 공간과 같은 자원을 공유하며) 결정됩니다. 아무런 플래그도 설정하지 않으면 `fork()`와 유사하게 작동합니다.
