# 운영체제

운영체제도 하나의 프로그램입니다. 따라서 운영체제 또한 여느 프로그램과 마찬가지로 메모리에 적재되어야 합니다.

하지만 운영체제는 특별한 프로그램이기 때문에 **커널 영역**(kernel space)이라는 공간에 적재되어 실행됩니다. 이 커널 영역을 제외한 나머지 영역을 **사용자 영역**(user space)이라고 합니다. 바로 이 영역에 우리가 사용하는 각종 응용 프로그램이 적재됩니다.

운영체제는 프로그램이 실행되어 메모리에 적재될 때 어떤 프로그램을 어느 메모리에 적재할 지 정하고 종료된 프로그램을 메모리에서 제거하는 역할을 합니다. 프로그램이 실행되는 데 필수 요소인 CPU에 프로그램을 할당하는 것 역시 운영체제가 담당합니다.

## 커널 (Kernel)

커널은 자원에 접근하고 조작하는 기능, 프로그램이 올바르고 안전하게 실행되게 하는 기능 등의 운영체제 핵심 서비스를 담당합니다.

## 이중모드 (Dual Mode)

운영체제는 사용자가 실행하는 응용 프로그램이 자원에 직접 접근하는 것을 방지하여 자원을 보호합니다. 만약 응용 프로그램이 CPU, 메모리, 하드 디스크 등에 마음대로 접근하고 조작할 수 있다면 자원이 무질서하게 관리될 것이고, 응용 프로그램에서 발생한 아주 작은 실수로 인해 컴퓨터 전체에 악영향을 끼칠 수 있습니다.

따라서 운영체제는 응용 프로그램들이 자원에 접근하려고 하면 오직 자신을 통해서만 접근하도록하여 자원을 보호합니다. 응용 프로그램의 요청을 받은 운영체제는 응용 프로그램 대신 자원에 접근하여 요청한 작업을 수행합니다.

이러한 운영체제의 문지기 역할은 이중 모드로써 구현됩니다. **이중 모드**란 CPU가 명령어를 실행하는 모드를 크게 **사용자 모드**(user mode)와 **커널 모드**(kernel mode)로 구분하는 방식입니다. CPU는 명령어를 사용자 모드로써 실행할 수 있고, 커널 모드로써 실행할 수 있습니다.

사용자 모드는 운영체제 서비스를 제공받을 수 없는 실행 모드입니다. 즉, 커널 영역의 코드를 실행할 수 없는 모드입니다. 일반적인 응용 프로그램은 기본적으로 사용자 모드로 실행됩니다. 사용자 모드로 실행 중인 CPU는 입출력 명령어와 같이 하드웨어 자원에 접근하는 명령어를 실행할 수 없습니다.

커널 모드는 운영체제 서비스를 제공받을 수 있는 실행 모드입니다. 즉, 커널 영역의 코드를 실행할 수 있는 모드입니다. CPU가 커널 모드로 명령어를 실행하면 자원에 접근하는 명령어를 비롯한 모든 명령어를 실행할 수 있습니다. 운영체제는 커널 모드로 실행되기 때문에 자원에 접근할 수 있습니다.

## 시스템 호출 (System Call)

사용자 모드로 실행되는 프로그램이 자원에 접근하는 운영체제 서비스를 제공받으려면 운영체제에 요청을 보내 커널 모드로 전환되어야 합니다. 이때 운영체제 서비스를 제공받기 위한 요청을 **시스템 호출**이라고 합니다. 사용자 모드로 실행되는 프로그램은 시스템 호출을 통해 커널 모드로 전환하여 운영체제 서비스를 제공받을 수 있습니다.

시스템 호출은 일종의 인터럽트입니다. 정확히는 소프트웨어적인 인터럽트입니다. 인터럽트는 입출력장치에 의해 발생하기도 하지만 인터럽트를 발생시키는 특정 명령어에 의해 발생하기도 하는데, 이를 소프트웨어 인터럽트라고 합니다.

시스템 호출을 발생시키는 명령어가 실행되면 CPU는 지금까지의 작업을 백업하고 커널 영역 내에 시스템 호출을 수행하는 코드를 실행한 뒤 다시 기존에 실행하던 응용 프로그램으로 복귀하여 실행을 계속해 나갑니다.

## 프로세스

프로세스는 사용자가 볼 수 있는 공간에서 실행되는 프로세스인 **포그라운드 프로세스**(foreground process)와 사용자가 보지 못하는 뒤편에서 실행되는 프로세스인 **백그라운드 프로세스**(background process)가 있습니다.

백그라운드 프로세스 중에는 사용자와 상호작용하는 프로세스가 있고, 상호작용하지 않는 프로세스가 있습니다. 사용자와 상호작용하지 않는 백그라운드 프로세스를 유닉스 체계의 운영체제에서는 **데몬**(daemon)이라고 부르고, 윈도우 운영체제에서는 **서비스**(service)라고 합니다.

## 프로세스 제어 블록 (PCB, Process Control Block)

프로세스는 실행을 위해 CPU를 필요로 하지만 CPU 자원은 한정적입니다. 그렇기에 프로세스는 차례대로 돌아가며 한정된 시간만큼만 CPU를 이용합니다. 정해진 시간만큼 CPU를 이용하고 시간이 끝나면 타이머 인터럽트가 발생하며 자신의 차례를 양보하고 다음 차례가 올 때까지 기다립니다.

운영체제는 이러한 프로세스의 실행 순서를 관리하고 CPU 자원을 배분합니다. 이를 위해 운영체제는 **프로세스 제어 블록**을 이용합니다.

프로세스 제어 블록은 프로세스와 관련된 정보를 저장하는 자료 구조입니다. **PCB는 커널 영역에 생성됩니다.** 운영체제가 PCB를 통해 프로세스를 식별하고 처리하는 데 필요한 정보를 판단합니다. PCB는 프로세스 생성 시에 만들어지고 실행이 끝나면 폐기됩니다. 즉 '새로운 프로세스가 생성되었다'는 말은 '운영체제가 PCB를 생성했다'라는 말과 같습니다.

PCB에 담기는 정보는 **프로세스 ID**(PID, process ID), **레지스터 값**, **프로세스 상태**, **CPU 스케줄링 정보**, **메모리 관리 정보**, **사용한 파일과 입출력장치 목록**이 있습니다.

- **프로세스 ID**: 프로세스를 식별하기 위해 부여되는 고유번호입니다. 동일한 프로그램을 두 번 실행하면 서로 다른 PID가 부여됩니다.
- **레지스터 값**: 프로세스는 자신이 진행했던 작업들을 저장하고 복원합니다. 그래서 프로세스가 실행하며 사용했던 프로그램 카운터를 비롯한 레지스터 값들이 담깁니다.
- **프로세스 상태**: 현재 프로세스가 입출력장치를 기다리고 있는 상태인지, CPU를 기다리고 있는 상태인지, 아니면 CPU를 이용하고 있는 상태인지 등의 상태 정보가 저장됩니다.
- **CPU 스케줄링 정보**: 프로세스가 언제, 어떤 순서로 CPU를 할당받을지에 대한 정보도 저장됩니다.
- **메모리 관리 정보**: 프로세스가 메모리의 어디에 적재되어 있는지에 대한 정보가 있어야 합니다. 프로세스의 주소를 알기 위하여 베이스 레지스터, 한계 레지스터, 페이지 테이블 정보가 저장됩니다.
- **사용한 파일과 입출력장치 목록**: 프로세스가 실행 중 특정 입출력장치나 파일을 사용하면 PCB에 해당 내용이 명시됩니다.

## 컨텍스트 스위칭 (Context Switching)

CPU를 할당받아 실행되고 있는 프로세스가 시간이 다 되어 다음 프로세스에게 CPU 사용을 양보하면 어떻게 될까요?

기존에 실행되고 있던 프로세스는 프로그램 카운터를 비롯한 각종 레지스터 값, 메모리 정보, 실행을 위해 열었던 파일이나 사용한 입출력장치 등의 중간 정보를 백업합니다. 그래야 다음에 CPU 사용 차례가 왔을 때 이전에 실행하던 내용을 이어서 실행할 수 있습니다. 이러한 중간 정보, 즉 하나의 프로세스 수행을 재개하기 위해 기억해야 할 정보를 **컨텍스트**(문맥, context)라고 합니다. 이 컨텍스트는 해당 프로세스의 PCB에 표현되어 있습니다. PCB에 기록되는 정보들을 컨텍스트라고 봐도 무방합니다.

프로세스의 CPU 사용 시간이 다 되거나 예기치 못한 상황이 발생하여 인터럽트가 발생하면 운영체제는 해당 프로세스의 PCB에 문맥을 백업합니다. 그리고 그 다음 차례의 프로세스의 컨텍스트를 복구합니다. 이렇게 기존 프로세스의 컨텍스트를 PCB에 백업하고, 새로운 프로세스를 실행하기 위해 컨텍스트를 PCB로부터 복구하여 새로운 프로세스를 실행하는 것을 **컨텍스트 스위칭**(context switching)이라고 합니다. 컨텍스트 스위칭은 충분히 빠르기 때문에 우리가 볼 때 여러 프로세스들이 동시에 실행되는 것처럼 보입니다.

## 프로세스 메모리 영역

PCB는 커널 영역에 생성된다고 했습니다. 그리고 프로세스는 사용자 영역에 로드됩니다.

하나의 프로세스는 사용자 영역에 저장될 때 **코드 영역**, **데이터 영역**, **힙 영역**, **스택 영역**으로 나뉩니다.

- **코드 영역(code segment)**: **텍스트 영역**(text segment)이라고도 부릅니다. 여기에는 실행할 수 있는 코드, 즉 기계어로 된 명령어가 저장됩니다. CPU가 실행할 명령어가 담겨 있기 때문에 쓰기가 금지되어 있습니다. 즉, 코드 영역은 읽기 전용 공간입니다. 동일한 명령에 대해서 처음 한 번만 로드되고 모든 프로세스에서 공유됩니다. 코드 영역은 정적 할당 영역입니다.
- **데이터 영역(data segment)**: 전역 변수(global variables)가 저장됩니다. 데이터 영역은 정적 할당 영역입니다.
- **힙 영역(heap segment)**: 프로그램 실행 중에 동적으로 할당되는 영역입니다. 참조형 객체가 저장됩니다. 메모리 주소 공간에서 위쪽으로 할당됩니다.
- **스택 영역(stack segment)**: 함수 호출, 지역 변수(local variables), 함수 호출 시 반환 주소가 저장됩니다. 힙 영역과 마찬가지로 동적 할당 영역입니다. 메모리 주소 공간에서 아래쪽으로 할당됩니다.

## 프로세스 상태

프로세스가 가질 수 있는 상태는 아래와 같습니다.

- **생성 상태(new)**: 프로세스가 생성되는 중인 상태를 나타냅니다. 메모리에 적재되어 PCB를 할당받은 상태를 말합니다.
- **준비 상태(ready)**: CPU에 할당되기를 기다리는 상태입니다.
- **실행 상태(running)**: CPU를 할당받아 실행 중인 상태입니다. 실행 상태인 프로세스는 할당된 일정 시간 동안만 CPU를 사용할 수 있고, 할당된 시간이 다 되면 타이머 인터럽트가 발생하며 준비 상태가 됩니다. 또한 실행 도중 입출력장치를 사용하여 입출력장치의 작업이 끝날 때까지 기다려야 한다면 대기 상태가 됩니다.
- **대기 상태(blocked)**: 입출력장치를 이용한 작업은 CPU 처리 속도에 비해 느리기 때문에 입출력 작업을 요청한 프로세스는 입출력장치가 작업을 끝낼 때까지, 즉 입출력 완료 인터럽트를 받을 때까지 기다려야 합니다. 이 상태를 대기 상태라고 하며, 입출력 작업이 완료되면 준비 상태가 됩니다.
- **종료 상태(terminated)**: 프로세스가 실행을 마친 상태입니다. 프로세스가 종료되면 운영체제는 PCB와 프로스세가 사용한 메모리를 정리합니다. 다만, 부모 프로세스가 `wait()` 시스템 호출을 하기 전까지는 프로세스 테이블에 해당 프로세스의 항목이 남아 있는데, 이 상태의 프로세스를 좀비(zombie) 프로세스라고 부릅니다.

## 프로세스 계층 구조

프로세스는 실행 도중 시스템 호출을 통해 다른 프로세스를 생성할 수 있습니다. 새 프로세스를 생성한 프로세스를 **부모 프로세스**(parent process), 부모 프로세스에 의해 생성된 프로세스를 **자식 프로세스**(child process)라고 합니다. 이러한 구조를 트리 구조로 표현할 수 있는데, 이를 프로세스 계층 구조라고 합니다.

모든 프로세스의 가장 위에 있는 최초의 프로세스는 무엇일까요? 유닉스 운영체제에서는 `init`, 리눅스 운영체제에서는 `systemd`, macOS에서는 `launchd`라고 합니다.

## 프로세스 생성 기법

프로세스 생성은 운영체제가 새로운 실행 흐름을 시작하는 핵심적인 작업입니다.

### 유닉스

부모 프로세스는 `fork()`를 호출하여 자신의 복사본을 만들어 자식 프로세스를 생성합니다. 이때 부모 프로세스의 모든 메모리 영역과 열린 파일 목록, 레지스터 상태 등이 복사됩니다. 그리고 자식의 프로세스 ID를 반환받습니다. 자식 프로세스는 `exec()`를 통해 자신의 메모리 공간을 다른 프로그램으로 교체하여 완전히 새로운 프로세스로 변환합니다. 자식 프로세스가 `exec()`를 호출하지 않는 경우도 있습니다. 이 경우 부모 프로세스와 자식 프로세스는 같은 코드를 병행하여 실행하는 프로세스가 됩니다.

전통적으로 `fork()`는 부모의 메모리 공간을 자식에게 복사하는 방식으로 작동했지만 자식 프로세스 생성 직후 `exec()`를 호출한다는 점을 고려할 때 이러한 복사는 불필요할 수 있습니다. **Copy-on-Write**(COW)는 이러한 비효율성을 개선합니다. 이 기술은 부모와 자식 프로세스가 초기에 동일한 페이지를 공유하도록 허용합니다. 만약 부모나 자식 프로세스 중 하나가 공유된 페이지에 쓰기 작업을 시도하면 그때서야 해당 페이지의 복사본이 생성되어 각 프로세스가 자신만의 독립적인 페이지를 가질 수 있게 됩니다. 이를 통해 프로세스 생성 속도가 빨라지고 필요한 경우에만 메모리 복사 오버헤드가 발생하여 효율적입니다.

### 리눅스

리눅스에서는 `clone()`을 이용하여 프로세스 또는 스레드와 유사한 태스크를 생성합니다. `clone()`에 전달되는 플래그 세트에 따라 생성되는 태스크가 프로세스처럼 동작할지(자원 공유없이) 스레드처럼 동작할지(메모리 공간과 같은 자원을 공유하며) 결정됩니다. 아무런 플래그도 설정하지 않으면 `fork()`와 유사하게 작동합니다.

## 스레드

스레드는 프로세스의 실행의 단위입니다. 그리고 CPU가 실제로 스케줄링하는 기본 단위입니다.

프로세스는 한 개 이상의 스레드를 가질 수 있습니다. 그래서 스레드를 이용하여 하나의 프로세스에서 여러 스레드를 이용하여 동시에 여러 작업을 수행할 수 있습니다.

스레드는 실행에 필요한 최소한의 정보만을 유지한 채 프로세스 자원을 공유하며 실행됩니다.

- 공유 자원
  - 코드 영역
  - 데이터 영역
  - 힙 영역
- 독자적 자원
  - 스택 영역
  - 레지스터 세트: 프로그램 카운터, 스택 포인터 등 현재 실행 중인 스레드의 상태를 나타내는 중요한 정보를 담고 있습니다.
  - 스레드 로컬 저장소(TLS, Thread-Local Storage): 함수를 넘어 유지되는 데이터입니다. 전역 변수와 유사하지만, 각 스레드별로 복사본이 존재하여 스레드별로 독립적인 데이터를 가질 수 있게 합니다.

## 멀티스레딩 (Multithreading)

하나의 프로세스는 여러 스레드를 가질 수 있습니다. 이때 여러 스레드로 하나의 프로세스를 동시에 실행하는 것을 **멀티스레딩**(multithreading)이라고 합니다. 반면에 여러 프로세스를 동시에 실행하는 것은 멀티프로세스(multiprocess)라고 합니다.

멀티스레딩은 여러 가지 이점을 제공항여 현대 컴퓨팅 환경에서 필수적인 요소가 되었습니다.

- **응답성(Responsiveness)**: 응용 프로그램의 일부가 차단되거나 긴 작업을 수행하더라도, 다른 부분은 계속 실행될 수 있으므로 사용자 인터페이스가 응답성을 유지할 수 있습니다.
- **자원 공유**: 스레드는 속한 프로세스의 코드, 데이터, 파일 등 많은 자원을 공유합니다. 이는 여러 프로세스를 생성하는 것보다 효율적입니다.
- **경제성**: 스레드를 생성하고 컨텍스트 스위칭하는 것은 프로세스를 생성하고 컨텍스트 스위칭하는 것보다 훨씬 적은 자원을 필요로 합니다.
- **확장성(Scalability)**: 멀티코어 프로세스에서 여러 스레드가 병렬로 실행될 수 있으므로 응용 프로그램이 다수의 처리 코어를 활용하여 성능을 향상시킬 수 있습니다.

반면에 멀티스레딩은 다음과 같은 과제를 지니고 있습니다.

- **테스트 및 디버깅**: 단일 스레드 응응 프로그램보다 테스트 및 디버깅이 본질적으로 더 어렵습니다.
- **동기화(Synchronization)**: 여러 스레드가 공유 자원에 동시에 접근할 때 발생하는 **경쟁 상태**(race condition)를 피하기 위해 적절한 동기화 메커니즘이 필요합니다.

## CPU 스케줄링

운영 체제가 CPU 활용을 최대화하고 사용자에게 빠른 응답 시간을 제공하기 위해 여러 프로세스 및 스레드에 CPU 코어를 할당하는 것을 **CPU 스케줄링**이라고 합니다.

## 프로세스 우선순위

우선순위가 높은 프로세스란 빨리 처리해야 하는 프로세스들을 의미합니다. 우선순위가 높은 프로세스에는 대표적으로 입출력 작업이 많은 프로세스가 있습니다.

대부분의 프로세스들은 CPU와 입출력장치를 모두 사용하며 실행됩니다. 즉, 프로세스는 실행 상태와 대기 상태를 반복하며 실행됩니다.

프로세스 종류마다 입출력 장치를 이용하는 시간과 CPU를 이용하는 시간의 양에는 차이가 있습니다. 계산보다 입출력 작업이 많은 프로세스를 **입출력 집중 프로세스**(I/O bound process)라 하고, 입출력 작업보다는 계산에 더 많은 시간을 사용하는 프로세스를 **CPU 집중 프로세스**(CPU bound process)라고 합니다.

입출력 집중 프로세스와 CPU 집중 프로세스가 동일하게 있을 때 동일한 우선수위로 CPU를 사용하는 것은 비효율적입니다. 입출력 집중 프로세스는 CPU를 오래 점유하지 않기 때문에 가능한 빨리 실행시켜 입출력장치를 작동시키도록 하고 그 뒤에 CPU 집중 프로세스를 실행하는 것이 더 효율적이기 때문입니다.

운영체제는 각 프로세스의 PCB에 우선순위를 명시하고 그 우선순위를 기준으로 먼저 처리할 프로세스를 결정합니다. 이렇게 우선순위가 높은 프로세스는 더 빨리, 더 자주 실행됩니다.

## 스케줄링 큐

PCB에 우선순위가 적혀 있기는 하지만 CPU를 사용할 다음 프로세스를 찾을 때마다 모든 PCB를 조회하는 것은 효율적이지 못합니다. 그래서 운영체제는 프로세스들을 줄을 세웁니다. 이 줄을 **스케줄링 큐**(scheduling queue)로 구현하고 관리합니다.

- **준비 큐(Ready Queue)**: CPU를 이용하려고 기다리는 프로세스들이 위치하는 큐입니다. 프로세스가 준비(ready) 상태가 되면 이 큐에 추가됩니다.
- **대기 큐(Wait Queue)**: 특정 이벤트가 발생하기를 기다리는 프로세스들이 위치하는 큐입니다. 입출력장치를 위해 대기(blocked) 상태가 되면 이 큐에 추가됩니다.

운영체제는 큐에 삽입된 순서대로 프로세스를 하나씩 꺼내어 실행하지만 그중 우선순위가 높은 프로세스를 먼저 실행합니다. 우선순위가 낮은 프로세스는 큐에 먼저 삽입이 되었더라도 우선순위가 높은 프로세스가 그보다 먼저 처리될 수 있습니다.

## 선점형, 비선점형 스케줄링

**선점형 스케줄링**(preemptive scheduling)은 프로세스가 CPU를 비롯한 자원을 사용하고 있더라도 운영체제가 프로세스로부터 자원을 강제로 빼앗아 다른 프로세스에 할당할 수 있는 스케줄링 방식을 의미합니다. 프로세스마다 정해진 시간만큼 CPU를 사용하고 정해진 시간을 모두 소비하여 타이머 인터럽트가 발생하면 운영체제가 해당 프로세스부터 CPU 자원을 빼앗아 다음 프로세스에 할당하는 방식이 선점형 스케줄링의 일종으로 볼 수 있습니다. 어느 한 프로세스의 자원 독점을 막고 프로세스들에 골고루 자원을 배분할 수 있다는 장점이 있지만, 그만큼 컨텍스트 스위칭 과정에서 오버헤드가 발생할 수 있습니다.

**비선점형 스케줄링**(non-preemptive scheduling)은 하나의 프로세스가 자원을 사용하고 있다면 그 프로세스가 종료되거나 스스로 대기 상태에 접어들기 전까진 다른 프로세스가 끼어들 수 없는 스케줄링 방식을 의미합니다. 컨텍스트 스위칭 횟수가 선점형 스케줄링보다 적기 때문에 오버헤드는 적지만, 하나의 프로세스가 자원을 사용 중이라면 다른 프로세스는 무작정 기다려야 하는 상황이 발생합니다.

현대 운영 체제의 대부분은 선점형 스케줄링 알고리즘을 사용합니다.

## 스케줄링 알고리즘

- **선입선출(FCFS, First-Come First-Served) 스케줄링**: 준비 큐에 삽입된 순서대로 프로세스들을 처리하는 가장 간단한 스케줄링 알고리즘이며 비선점형 스케줄링 방식입니다. 평균 대기 시간이 상당히 길어질 수 있으며, CPU 버스트 시간이 큰 프로세스가 섞여 있을 때 긴 시간을 기다려야 하는 **호위 효과**(convoy effect)가 발생하여 CPU 활용률을 낮출 수 있습니다.
- **최단 작업 우선(SJF, Shortest-Job-First) 스케줄링**: 프로세스의 다음 CPU 버스트 길이에 따라 CPU를 할당합니다. CPU 사용 시간이 긴 프로세스는 나중에 실행하고 CPU 사용 시간이 짧은 간단한 프로세스를 먼저 실행하여 호위 효과를 방지합니다. SJF는 기본적으로 비선점형 스케줄링 알고리즘으로 분류되지만, 선점형으로도 구현될 수 있습니다. 선점형 SJF 스케줄링은 **최단 잔여 시간 우선 스케줄링**으로 불립니다.
- **라운드 로빈(RR, Round-Robin) 스케줄링**: FCFS 스케줄링과 유사하지만 선점 기능이 추가된 스케줄링 알고리즘입니다. 시간 퀀텀(time quantum) 또는 타임 슬라이스(time slice)라고 불리는 프로세스가 CPU를 사용할 수 있는 정해진 시간 동안 돌아가며 CPU를 이용할 수 있게 하는 선점형 스케줄링입니다. 큐에 삽입된 프로세스들은 순서대로 CPU를 이용하되 정해진 시간만큼만 CPU를 이용하고 그 시간 안에 완료되지 않았다면 다시 큐의 맨 뒤에 삽입되는 원형 큐의 형태로 동작합니다. 이때 컨텍스트 스위칭이 발생합니다. 라운드 로빈 스케줄링에서는 타임 슬라이스의 크기가 매우 중요합니다. 타임 슬라이스가 지나치게 크면 사실상 FCFS 스케줄링과 동일하여 호위 효과가 생길 여지가 있고, 타임 슬라이스가 지나치게 작으면 컨텍스트 스위칭 발생 비용이 커지게 됩니다.
- **최소 잔여 시간 우선(SRT, Shortest Remaining Time) 스케줄링**: 최단 작업 우선 스케줄링과 라운드 로빈 스케줄링을 합친 스케줄링 방식입니다. 프로세스들의 정해진 타임 슬라이스만큼 CPU를 사용하되 CPU를 사용할 다음 프로세스로는 남아있는 작업 시간이 가장 적은 프로세스가 선택됩니다.
- **우선순위(Priority) 스케줄링**: 프로세스들에 우선순위를 부여하고 가장 높은 우선순위를 가진 프로세스부터 실행하는 스케줄링 알고리즘입니다. 우선순위가 같은 프로세스는 선입선출 스케줄링으로 처리됩니다. 우선순위가 높은 프로세스를 우선하여 처리하는 방식이기에 우선순위가 낮은 프로세스는 계속해서 연기될 수 있습니다. 이를 **기아**(starvation) 현상이라고 합니다. **에이징**(aging) 기법을 사용하여 이 문제를 해결할 수 있습니다. 오랜 시간 기다린 프로세스의 우선순위를 점진적으로 높여 CPU를 할당받을 수 있도록 하는 것입니다.
- **다단계 큐(Multilevel Queue) 스케줄링**: 우선순위 스케줄링의 발전된 형태로, 우선순위별로 준비 큐를 여러 개 사용하는 스케줄링 알고리즘입니다. 우선순위가 가장 높은 큐에 있는 프로세스들을 먼저 처리하고, 우선순위가 가장 높은 큐가 비어 있으면 그다음 우선순위 큐에 있는 프로세스들을 처리합니다. 프로세스들이 큐 사이를 이동할 수 없습니다. 큐 별로 스케줄링 알고리즘을 다르게 선택할 수 있고, 타임 슬라이스 역시 다르게 적용할 수도 있습니다.
- **다단계 피드백 큐(Multilevel Feedback Queue) 스케줄링**: 다단계 큐 스케줄링의 발전된 형태로, 프로세스들이 큐 사이를 이동할 수 있도록 허용하여 기아 현상을 방지하며 다단계 큐 스케줄링의 유연성을 높입니다.

## 프로세스 동기화

프로세스 동기화는 여러 프로세스(또는 스레드)가 공유하는 데이터에 동시에 접근할 때 데이터의 일관성을 유지하고, 프로세스들의 실행 순서를 제어하여 발생할 수 있는 문제(경쟁 조건, 교착 상태)를 방지하는 메커니즘을 의미합니다.

프로세스 동기화가 필요한 주된 이유는 **경쟁 조건**(race condition) 때문입니다. 여러 프로세스가 공유 자원에 동시에 접근하여 문제가 발생하는 상황을 경쟁 조건이라고 합니다. 경쟁 조건이 발생하는 이유는 연산이 원자적(atomic)으로 수행되지 않고 중간에 다른 프로세스에 의해 끼어들 수 있기 때문에 발생합니다. 이러한 문제를 해결하려면 공유 자원에 접근하는 코드 영역, 즉 **임계 구역**(critical section)에 한 번에 하나의 프로세스만 접근하도록 보장해야 합니다.

임계 구역 문제 해결을 위한 세 가지 요구 사항은 다음과 같습니다.

- **상호 배제(Mutual Exclusion)**: 한 프로세스가 임계 구역에 있으면 다른 프로세스는 임계 구역에 들어갈 수 없습니다.
- **진행(Progress)**: 임계 구역에 아무도 없고, 임계 구역에 들어가려고 하는 프로세스가 있을 때, 다음에 임계 구역에 들어갈 프로세스를 유한한 시간 안에 선택해야 합니다.
- **한정된 대기(Bounded Waiting)**: 임계 구역에 들어가려고 하는 프로세스가 무한히 기다리지 않도록 보장해야 합니다.

## 동기화 기법

- **뮤텍스 락(Mutex Lock)**: 가장 기본적인 동기화 도구로, 상호 배제를 구현하는 데 사용됩니다. 가장 단순한 형태는 하나의 전역 변수와 두 개의 함수로 구현할 수 있습니다. 가용 여부를 나타내는 불리언 `available` 전역 변수를 가지며, 임계 구역을 잠그는 역할인 `acquire()` 함수, 임계 구역의 잠금을 해제하는 `release()` 함수가 있습니다. `acquire()` 호출 시 락을 획득하고 `available`을 false로 설정합니다. 사용 불가능한 락을 획득하려는 프로세스는 락이 해제될 때까지 블록(대기)됩니다. `release()` 호출 시 락을 해제하고 `available`을 true로 설정합니다.
- **세마포(semaphore)**: 뮤텍스 락과 비슷하지만 조금 더 일반화된 방식의 동기화 도구입니다. 뮤텍스 락은 하나의 공유 자원에 접근하는 프로세스에 대한 방식인데 반해, 세마포는 공유 자원이 여러 개 있는 상황에서도 적용이 가능한 동기화 도구입니다. 임계 구역에 진입할 수 있는 프로세스의 개수(사용 가능한 공유 자원의 개수)를 나타내는 전역 변수 `S`와 임계구역에 들어가도 좋은지, 기다려야 할지를 알려주는 `wait()` 함수, 임계 구역 앞에서 기다리는 프로세스에 '이제 가도 좋다'고 신호를 주는 `signal()` 함수로 구현할 수 있습니다. `wait()` 함수는 호출 시 `S` 값이 0보다 작거나 같으면 프로세스는 바쁜 대기(busy wait) 상태로 대기하고 그렇지 않으면 `S` 값을 1 감소시킵니다. `signal()` 함수는 `S` 값을 1 증가시킵니다. 변수 `S`는 음수가 될 수 있으며 음수 값의 크기는 해당 세마포에서 대기 중인 프로세스 수를 나타냅니다.
- **모니터(Monitor)**: 공유 데이터와 해당 데이터를 조작하는 프로그래머 정의 연산을 캡슐화한 추상 데이터 타입(ADT)입니다. 모니터 내의 연산은 상호 배제를 제공합니다. 특정 조건을 바탕으로 프로세스를 실행하고 일시 중단하기 위해 모니터는 **조건 변수**(condition variable)을 사용합니다. 조건 변수는 프로세스나 스레드의 실행 순서를 제어하기 위해 사용하는 특별한 변수입니다. 조건 변수로 `wait()` 함수와 `signal()` 함수를 사용할 수 있습니다. `x.wait()`는 해당 프로세스를 중단시키고, `x.signal()`은 중단된 프로세스 중 하나를 다시 시작합니다.

## 교착 상태(Deadlock)

운영 체제에서 여러 프로세스 또는 스레드가 서로 보유한 자원을 기다리며 무한히 블록되는 상황을 의미합니다.

## 식사하는 철학자 문제

교착상태를 설명하기 위한 고전적인 문제 상황입니다. 둥그런 원탁에 다섯 명의 철학자가 앉아 있습니다. 이 철학자들 앞에는 음식이 있고 철학자들 사이에는 포크가 있습니다. 철학자들은 식사를 하기 위해서 두 개의 포크가 필요한 상황이라고 가정합니다. 왼쪽 포크가 사용가능하면 집어 들고, 오른쪽 포크도 사용가능하면 집어 듭니다. 이렇게 두 개의 포크를 모두 집어 들면 식사를 할 수 있습니다. 그런데 모든 철학자들이 자신의 왼쪽에 있는 포크를 집어 들게 되면 남아 있는 포크가 없게 되고 모두가 오른쪽 포크를 사용하기 위해 기다리게 됩니다. 이렇게 다른 자원을 기다리며 진행이 멈춰버리는 현상을 교착 상태라고 합니다.

## 자원 할당 그래프

자원 할당 그래프는 어떤 프로세스가 어떤 자원을 사용하고 있고, 어떤 프로세스가 어떤 자원을 기다리고 있는지를 표현하는 간단한 그래프입니다. 프로세스는 원으로 자원의 종류는 사각형으로 표현합니다. 프로세스가 자원을 사용 중이면 자원에서 프로세스를 향해 화살표를 표시합니다. 프로세스가 자원을 기다리는 중이면 프로세스에서 자원으로 화살표를 표시합니다.

만약 교착상태가 발생하면 프로세스와 자원들 사이에서 화살표가 원 형태(사이클)를 이루게 됩니다.

## 교착 상태 발생 조건

교착 상태가 발생하는 조건에는 네 가지가 있습니다. 이 네 가지를 모두 충족해야 교착 상태가 발생하게 됩니다.

- **상호 배제**: 적어도 하나의 자원은 공유 불가능한 방식으로 보유되어야 합니다. 즉, 한 번에 하나의 프로세스만이 자원을 사용할 수 있습니다.
- **점유와 대기(Hold and Wait)**: 자원을 적어도 하나 보유한 프로세스가 다른 자원을 얻기 위해 대기하고 있어야 합니다. 이 다른 자원은 이미 다른 프로세스에 의해 점유되어 있습니다.
- **비선점(No Preemption)**: 자원은 선점될 수 없습니다. 즉, 자원을 보유한 프로세스에 의해서만 자원 점유가 해제될 수 있습니다.
- **순환 대기(Circular Wait)**: 프로세스들과 프로세스가 요청 및 할당받은 자원이 원의 형태를 이루는 것을 말합니다. 즉, {P0, P1, ..., Pn} 집합의 프로세스가 존재할 때, P0은 P1이 보유한 자원을 기다리고, P1은 P2가 보유한 자원을 기다리고, ..., Pn-1은 Pn이 보유한 자원을 기다리고, Pn은 P0가 보여한 자원을 기다리는 형태입니다.

## 교착 상태 해결 방법

- **교착 상태 예방(Deadlock Prevention)**: 교착 상태의 네 가지 필요 조건 중 하나 이상이 발생하지 않도록 하여 교착 상태를 방지하는 방법입니다. 자원 요청이 이루어지는 방식을 제한하여 발생시킬 수 있습니다. 하지만 이러한 방법으로 교착 상태를 방지하면 자원 활용도가 낮아지고 시스템 처리량이 감소하는 부작용이 발생할 수 있습니다.
- **교착 상태 회피(Deadlock Avoidance)**: 스레드가 자원을 요청할 때, 시스템이 안전 상태(safe state)를 유지할 수 있는 경우에만 자원 할당을 허용합니다. 안전 상태는 교착 상태가 발생하지 않고 모든 프로세스가 정상적으로 자원을 할당받고 종료될 수 있는 상태를 말합니다. 반대로 교착 상태가 발생할 수도 있는 상황을 불안전 상태(unsafe state)라고 부릅니다.
- **교착 상태 탐지(Deadlock Detection)**: 시스템이 교착 상태에 들어갈 수 있도록 허용하고, 그 후에 탐지하여 복구하는 방법입니다. 복구 방법에는 **선점을 통한 회복**과 **프로세스 강제 종료를 통한 회복**이 있습니다. 선점을 통한 회복은 교착 상태가 해결될 때까지 한 프로세스씩 자원을 몰아주는 방식입니다. 프로세스 강제 종료를 통한 회복은 가장 단순하면서 확실한 방법입니다. 운영체제는 교착 상태에 놓인 프로세스를 모두 강제 종료할 수도 있고, 교착 상태가 없어질 때까지 한 프로세스씩 강제 종료할 수 있습니다.
